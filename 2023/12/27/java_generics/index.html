泛型的类型擦除

我们常看到这样一句话：“Java的泛型是伪泛型”。何为“伪泛型”？

“伪泛型”即Java在语法上支持泛型，但是在编译阶段会进行所谓的“类型擦除”（Type Erasure），将所有的泛型表示（尖括号中的内容）都替换为具体的类型（其对应的原生态类型），就像完全没有泛型一样。

Java泛型这个特性是从JDK 1.5才开始加入的，因此为了兼容之前的版本，Java泛型的实现采取了“伪泛型”的策略。

泛型的类型擦除原则是：

  • 消除类型参数声明，即删除<>及其包围的部分。
  • 根据类型参数的上下界推断并替换所有的类型参数为原生态类型：如果类型参数是无限制通配符或没有上下界限定则替换为Object，如果存在上下界限定则根据子类替换原则取类型参数的最左边限定类型（即父类）。
  • 为了保证类型安全，必要时插入强制类型转换代码。
  • 自动产生“桥接方法”以保证擦除类型后的代码仍然具有泛型的“多态性”。

ps：其他语言泛型实现方式

  • C# 的泛型不同于 Java 的类型擦除机制，C# 在运行时保留了类型信息。这意味着可以在运行时获取到具体的泛型类型和元信息。这种特性使得 C# 可以利用反射和泛型约束，提供更强的类型安全。
  • C++ 的泛型底层实现主要是通过模板（Templates），这在语言设计中是一个极其强大和灵活的特性。模板允许编写不针对特定类型的代码，而是在编译时生成适用于多种类型的代码。C++ 的模板在编译时展开，这意味着编译器为每一个使用模板的类型实例生成独立的代码。
  • 在 Go 的泛型实现中，每个类型都有一个对应的类型码，这使得有多个类型参数的函数或数据结构能够在运行时通过类型码进行适当的处理。这一机制类似于其他语言中的类型擦除，但 Go 的泛型实现会在编译时解析类型，使得类型信息能够在运行时进行适当处理。

其实从与其他语言比较看出来，Java泛型实现机制显得臃肿且复杂，这也是Java设计团队为了“兼容”而做出的取舍。

可是由于种种原因，虚拟机并不能将泛型类型变为Date，只能将类型擦除掉，变为原始类型Object。这样，我们的本意是进行重写，实现多态。可是类型擦除后，只能变为了重载。这样，类型擦除就和多态有了冲突。JVM知道你的本意吗？知道！！！可是它能直接实现吗，不能！！！如果真的不能的话，那我们怎么去重写我们想要的Date类型参数的方法啊。

后记与吐槽

Java的“伪泛型”策略与其他语言相比真不见的优秀，在编程上也增加了额外的限制，例如异常中使用泛型，这在Java上想都不敢想。事实上，泛型类连拓展Throwable都不合法，更别想在抛出或捕获泛型了。

有时候兼容是对产品负责任的表现，但有时处处兼容会导致产品后续设计处处受限，适当抛弃一些落后时代太多的版本而提高性能或体验也未尝不可。

Java总是抱着“祖宗之法不可变”的心态设计，当断不断必受其乱。隔壁c#大刀阔斧重构，.NET Core 3.0大获成功，Java也该考虑大刀阔斧的改革一番了。
