上古时期的垃圾回收器们

Serial 垃圾回收器

Serial垃圾回收器是单线程回收器，在GC时用户线程需要停止工作。其衍生出了Serial old垃圾回收器。

为什么GC时用户线程需要停止工作？

假设堆内有A、B对象，GCRoots引用A对象，A对象引用B对象，GC回收器工作时会标记哪些对象不被回收，因此A、B被标记为无需被回收，但GC时若程序又生成了一个C对象，且由B对象所引用。由于这个C对象没有被标记，它会被GC掉，当程序下次想要引用C时发现C对象不见了，直接down掉了。🤣

Parallel Scavenge

该死的，单线程效率不行，那我就上多线程🤬。这时就有了Parallel Scavenge。

Parallel Scavenge是多线程回收器，但在GC时同样需要用户线程停止工作。它是以吞吐量优先的垃圾回收器。

但是Parallel Scavenge仅仅适用于新生代。

这里的吞吐量指的是CPU用于运行用户代码的时间占总时间的比值。

CMS垃圾回收器

总是让用户线程停止，效率永远上不去，咋整。这时有了CMS垃圾回收器。

CMS垃圾回收器是第一款能够与用户线程并发执行的垃圾回收器。使用标记清除算法(名字都叫并发标记清除回收器了)。过程分为初始标记(串)、并发标记(并)、重新标记(串)与并发清除(并)。

CMS仅适用于老年代。

CMS为啥只适用于老年代？

很简单，新生代仅适用标记-复制算法。

看起来CMS+Parallel Scavenge是天作之和，但其实二者不共戴天，不可能一起使用。

ParNew 垃圾回收器

由于CMS与Parallel Scavenge不能一起使用，为了配合CMS推出了ParNew垃圾回收器与ParNew old。

ParNew垃圾回收器是Serial垃圾回收器的多线程版本。

新时代的垃圾回收器们

天外来客-G1垃圾回收器

G1垃圾回收器的画风显然与其他三种垃圾回收器完全不同。

G1的主要设计目标就是保证STW的时间和分布都可以很好的预测和配置. 事实上, Garbage-First 是一个软实时GC. 也就是你可以设定特定的性能要求. 你可以要求在给定的y ms中, STW的时间不能超过x ms. 比如在任何1s内都不超过5ms. G1 会尽量满足设定的目标(但不能完全肯定, 所以不是绝对实时的)。

G1垃圾回收器的目标：软实时、低延迟、可设定目标。

现在G1垃圾回收器是JDK9的默认GC，它适用于较大的堆(>4-6G)，用于替代CMS(最新几个版本的JDK中已经移除CMS了)。

为了达到这个目标, G1建立在大量的见解之上. 首先, 堆不再需要被分割进连续的2个年轻代和老年代. 取而代之的是, 堆被分为很多(典型的是2048)个小的*堆区域(heap regions)*来存储对象. 每个region都可能是Eden region, 或者Survivor region, Old region. 所有Eden和Survivor 区组合成了逻辑上的年轻代, 所有的Old region组合在一起成了老年代:

这让垃圾收集器不需要每次都收集整个堆,而是每次增量的解决问题:每次只会有所有region集合的一个子集会被考虑, 称为收集集合Collection set.年轻代的所有region在每个暂停的时候都被收集,但是老年代只有一部分会被收集:

在并发阶段的另一个新奇的事就是G1会估计每个region中包含的存活对象的个数.这被用来构建Collection set:包含最多垃圾的region总是优先被收集. 这个是名称Garbage-first的由来.

G1内部细节

  • 无需回收整个堆，而是选择一个Collection Set进行回收。
  • G1仅有两种回收方式，Fully young GC 与 Mixed GC。
  • 估计每个Region中的垃圾比例，优先回收垃圾多的Region。

一个Region又被分为一个个Card，Card Table每个entry覆盖512Byte的内存空间。当对应的内存空间发生改变时，标记为dirty。

一个Region对应一个RememberedSet，其指向引用该Region中Card的对象的对象。当对这个对象进行GC时，就可以根据RS找到引用它的对象(主要是为了解决跨代引用问题)。RememberedSet是用C语言写的HashTable。

以上两个结构，都是用时间换空间，用额外的空间维护引用信息。

前面提到了RS，那不得不提对RS信息的维护。

G1采用Write barrier与四色队列的方式维护RS信息。

  • Write barrier：每次对对象引用后，JVM都会注入一小段代码，用于记录指针的变化。

    object.field = (putfield)。但这一行为并不是立即更新RS，而是将这个对象放入四色队列中。

  • 四色队列：当更新指针时，标记Card为Dirty，将Card存入Dirty Card Queue。这个队列有白/绿/黄/红四种颜色。

    当队列处于白色时，天下太平，无事发生。

    当队列处于绿色时，Refinement线程开始激活，更新RS。

    当队列处于黄色时，全部Refinement线程激活。

    当队列处于红色时，应用线程也参与到更新RS的工作中。

Fully young GC

在引用的开始阶段, G1不能从还没执行过的并发阶段知道任何额外的信息, 所以它最开始工作在fully-young模式.当年轻代满了后,应用线程被暂停,年轻代regions的活跃对象被拷贝到Surivior regions(任何其他空闲的region因此就变成了Survivor).

年轻代的GC永远是标记-复制算法。

拷贝的过程称为Evacuation(译注:意为疏散). 这与我们前面讲到的年轻代的收集器完全一样.Evacuation Pause的GC 日志非常长, 所以这里简单期间, 我们只留下了一些与第一次fully-young Evacuation Pause相关的日志. 我们随后讲到并发阶段时还会详细讲到. 除此之外, 因为日子很大, 并发阶段和其他Other阶段会在单独的段落讲到:

  1. 0.134: [GC pause (G1 Evacuation Pause) (young), 0.0144119 secs] G1暂停只清理年轻代的region. 这个暂停在JVM启动后134ms开始,暂停时间0.0144s.
  2. [Parallel Time: 13.9 ms, GC Workers: 8] 表示有13.9ms (real time)用于后面的8个线程的GC活动
  3. … 简单起见,省略了, 后面会讲到
  4. Code Root Fixup: 0.0 ms 用于释放管理并发活动的数据结构, 该值应该总是0. 这是顺序执行的.
  5. [Code Root Purge: 0.0 ms] 清理更多的数据结构, 应该会很快. 但不一定是0. 这是顺序执行的.
  6. [Other: 0.4 ms] 混杂的其他活动时间, 其中的许多也是并行的.
  7. … 后面会讲到
  8. [Eden: 24.0M(24.0M)->0.0B(13.0M) 暂停前后Eden的使用量和容量
  9. *Survivors: 0.0B->3072.0K * 暂停前后, Survivor的使用量.
  10. Heap: 24.0M(256.0M)->21.9M(256.0M)] 暂停前后, 堆得使用量和容量.
  11. *[Times: user=0.04 sys=0.04, real=0.02 secs]*GC持续时间, 分为如下3类: - user GC线程在GC期间所用的总的CPU时间 - sys 系统调用和等待系统事件的时间 - real 应用线程总的停顿时间. 该值应大致等于 (user+system)/GC所用线程数. - 在这个例子中,使用了8个线程. 因为有些活动没有被并行化, 所以真实值总是比计算的值大一点.

Fully young GC过程可以被概况为：

  • STW
    • 构建CS(Eden+Survivor)
    • 扫描GC Roots
    • Update RS：排空Dirty Card Queue
    • Process RS：找到哪些对象被老年代引用
    • Object Copy
    • Reference Processing

Fully young GC需要注意的细节：

  • G1会记录每个阶段的时间，用于自动调优。
  • 记录Eden/Survivor的数量和GC时间。
    • 根据暂停目标自动调整Region的数量。
    • 暂停目标越短，Eden数量越少。(这样带来的后果是吞吐量下降)

Mixed GC(Old GC)

当堆用量达到一定程度时触发，可通过-XX:InitiatingHeapOccupancyPercent=N修改比例，默认是45%。Old GC是并发进行的，使用三色标记算法(在不暂停应用线程的情况下进行标记)

但是这种算法会带来Lost Object Problem问题。

这样会导致C没有被标记到，最后被清理掉。

因此G1采用Snapshot-At-The-Beginning (SATB)的方式保存marking阶段开始时的快照，即使C的指针被删除，仍然认为C是活对象，C仍可以被remark阶段处理。当然这样也会带来浮动垃圾的问题。

G1 Old GC流程

  • STW(Fully young GC)
    • piggy-backed on young GC
    • 利用Young GC的信息
  • 恢复应用线程
  • 并发初始标记
  • STW
    • Remark
      • SATB
    • Cleanup
      • 立刻回收全空区域
  • 恢复应用线程。

Mixed GC需要注意的细节

  • 不一定立即发生
  • 选择若干个Region进行
    • 默认1/8的Old Region
      • 可通过修改-XX:G1MixedGCCountTarget=N修改
    • Eden+Survivor Region
    • SWT，Parallel,Copying
  • 根据暂停目标，选择垃圾最多的Old Region优先进行
